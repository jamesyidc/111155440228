🚨 重要修复合集（2026-02-19）

🐛 问题1：保证金计算逻辑错误（提交: 753da18）

  • 现象：设置单笔保证金5U，但实际只占用0.5U保证金
  • 根因：前端传递 sz = marginPerCoin（5），后端理解为"合约价值5U"，实际保证金 = 5/10 = 0.5U
  • 用户需求：保证金5U × 10倍杠杆 = 50U合约价值

✅ 解决方案：

  • 前端修正：sz = contractValuePerCoin = marginPerCoin × 10（传50而不是5）
  • 计算流程：保证金5U → 合约价值 = 5×10 = 50U → sz=50 → OKX占用保证金 = 50/10 = 5U ✅
  • 增强日志：控制台详细显示每一步计算（保证金、合约价值、实际传值）
  • 影响范围：batchOrderTop8、batchOrderBottom8、batchOrder

🐛 问题2：自动策略反复触发（提交: 9b808c1）

  • 现象：策略执行后，每60秒又再次触发执行
  • 根因：用户打开开关时，前端立即写入 allowed=true 到JSONL → 执行后虽然写入 allowed=false，但用户再次打开开关又写入 allowed=true → 定时器检查发现 allowed=true → 再次触发
  • 触发流程：打开开关 → allowed=true → 触发执行 → allowed=false → 用户再开开关 → allowed=true → 60秒后再次触发

✅ 解决方案：

  • 删除自动写入：打开开关时不再自动写入 allowed=true
  • allowed管理规则：只能通过以下方式修改：①策略执行成功后自动设为false ②用户手动点击"重置执行许可"按钮设为true
  • 正确流程：打开开关 → 点击"🔄重置"按钮 → allowed=true → 等待触发 → 执行一次 → allowed=false

🐛 问题3：业务逻辑缺陷 - 缺少重置入口（提交: 789d788）

  • 现象：策略执行后，用户打开开关但策略不执行
  • 根因：执行后 allowed=false + enabled=false → 用户打开开关 enabled=true，但 allowed 仍是 false → 不满足执行条件（需要 enabled && allowed 都为true）
  • 用户困惑：开关已打开，为什么策略不执行？因为缺少重置 allowed 的UI入口

✅ 解决方案：

  • 添加"🔄重置"按钮：在"📄JSONL执行许可"状态旁边添加重置按钮
  • 重置流程：点击按钮 → 弹出确认框（显示账户、策略、触发价格）→ 确认后设置 allowed=true → 刷新状态显示
  • 完整使用流程：①打开策略开关 ②点击"🔄重置"按钮 ③等待BTC价格触发 ④策略自动执行一次
  • 再次使用：必须重复步骤①②，确保每次执行都是有意为之，防止误触发

🐛 问题5：XRP等大面值合约开仓金额严重超额（提交: 22be347）

  • 现象：XRP每次开仓金额都不对，目标5U保证金（50U合约价值），实际开出14.13 USDT保证金（140U合约价值），超额2.8倍
  • 受影响币种：XRP(100/张)、DOGE(1000/张)、ADA(100/张)、TRX(1000/张) - 所有大面值合约
  • 根因分析：
    • ① XRP真实合约规格：ctVal=100 XRP/张，每张≈140 USDT（价格1.40）
    • ② 目标开仓50 USDT → 需要 50/140 = 0.36张
    • ③ 致命Bug：代码强制 max(1, round(0.36)) = 1张
    • ④ 实际开仓：1张 = 140 USDT合约价值 = 14 USDT保证金 ❌
  • 错误代码：contracts_count = max(1, round(...)) # 强制至少1张
  • 误解根源：OKX支持小数张数（minSz=0.01），不需要强制取整为1张

✅ 解决方案：

  • 移除强制逻辑：删除 max(1, ...)，改为 round(contracts_count, 2)（保留2位小数）
  • 支持小数张数：0.36张 → 实际开仓50.4 USDT ✅（误差仅0.8%）
  • 最小下单量：如果计算结果 < 0.01张，则使用0.01张（而不是1张）
  • 修复后效果：
    • 目标50 USDT → 计算0.36张 → 实际开仓：0.36×140 = 50.4 USDT ✅
    • 目标5 USDT → 计算0.036张 → 实际开仓：0.04×140 = 5.6 USDT ✅
    • 超额倍数：从2.8倍降低到1.01倍（±1%误差）

⚠️ 危险性说明：

  • XRP：1张=100 XRP，强制1张 → 超额2.8倍
  • DOGE：1张=1000 DOGE，强制1张 → 超额10倍+
  • 风险等级：🔴 致命级（可能导致爆仓或重大亏损）

🐛 问题6：批量开仓风控检查失效（提交: 8bec4cb）

  • 现象：配置单笔保证金上限5 USDT，但实际开仓金额超过限制
  • 根因：前端传递 maxOrderSize: contractValuePerCoin（50 USDT）给后端，而不是账户配置的5 USDT
  • 后端检查：if 50 > 50: reject() → False，风控失效

✅ 解决方案：

  • 修正传值：改为 maxOrderSize: maxOrderSize × 10（5×10=50 USDT）
  • 风控生效：如果尝试开60 USDT → if 60 > 50: reject() → True ✅

🔒 当前风控机制（2026-02-20 更新）：

  • 保证金限额：单笔保证金≤5 USDT（可配置），10倍杠杆 = 50 USDT合约价值
  • 前端验证：marginPerCoin = Math.min(余额×1.5%, maxOrderSize)
  • 后端验证：服务器检查 sz（合约价值）是否超过 maxOrderSize×10
  • ✅ 小数张数支持：OKX允许0.01张起步，不再强制1张，避免大面值合约超额
  • ✅ 精确下单：保留2位小数（如0.36张），开仓金额误差控制在±1%以内
  • 双重状态控制：enabled（UI开关）+ allowed（JSONL执行许可），两者都为true才执行
  • 防重复机制：执行后立即设置 allowed=false，必须手动重置才能再次执行

⚠️ 重要提示：如遇到问题，请硬刷新浏览器（Ctrl+Shift+R）清除缓存，或使用隐身模式访问。

📌 升级亮点（2026-02-17）

本次升级解决了三个关键问题，确保策略执行完全独立，避免误触发和状态混乱。

✅ 修复1: 上涨占比实时显示 (v2.3)

  • 问题：上涨占比显示延迟或隐藏，需等待60秒或开关触发才显示
  • 原因：代码逻辑错误，将数据获取放在开关检查之后
  • 解决：将获取并显示上涨占比的代码提前，确保不论开关状态均实时显示
  • 效果：上涨占比始终可见，0%时红色高亮（#ef4444），>0%时灰色显示（#6b7280），60秒自动更新

✅ 修复2: 账户独立策略开关 (v2.4)

  • 问题：4个账户（main, fangfang12, poit, marks）共用同一套策略开关，切换账户后开关状态不更新
  • 原因：前端开关是全局单一DOM元素，未实现账户切换时重新加载对应账户的策略状态
  • 解决：新增 loadUpRatio0StrategySettings() 函数，在账户切换和页面初始化时调用，从API加载当前账户的策略状态并更新开关UI
  • 效果：每个账户的策略开关完全独立，切换账户自动加载对应状态，UI与JSONL文件保持一致

✅ 修复3: BTC策略JSONL分离 (v2.5)

  • 问题：BTC涨幅前8名和涨幅后8名共用同一个JSONL文件，启动其中一个会影响另一个
  • 原因：后端API使用 {account_id}_execution.jsonl 作为两个策略的共享文件
  • 解决：将BTC策略API修改为接受 strategy_type 参数，使用独立JSONL文件：
    account_{account_id}_btc_top_performers_execution.jsonl
    account_{account_id}_btc_bottom_performers_execution.jsonl
  • 效果：4个账户 × 4个策略（BTC-top8, BTC-bottom8, upratio0-top8, upratio0-bottom8）= 16个独立JSONL文件，完全避免冲突

  • 🆕 上涨占比0策略：每个账号、每个策略独立的执行许可文件：
    • data/okx_auto_strategy/account_{account_id}_upratio0_top8_execution.jsonl
    • data/okx_auto_strategy/account_{account_id}_upratio0_bottom8_execution.jsonl
  • 记录格式：{"allowed": true/false, "reason": "...", "timestamp": "...", "account_id": "...", "strategy_type": "..."}
  • 执行前通过API检查文件最后一条记录的 allowed 状态
  • 执行后立即通过API写入 allowed: false，阻止重复触发
  • ✅ 跨设备、跨浏览器、跨会话有效 - 这是唯一真相源

10.2 第二层：冷却期（时间层）

  • 5分钟强制冷却期，防止短时间内重复执行
  • 检查 lastExecutedTime，距离上次执行不足5分钟则拒绝

10.3 第三层：执行锁（进程层）

  • 全局变量 strategyExecuting 标记执行中状态
  • 同一浏览器页面内防止并发执行

10.4 工作流程

  1. 用户开启开关 → POST /api/okx-trading/set-allowed/{account_id} 写入 allowed: true
  2. 触发条件满足 → GET /api/okx-trading/check-allowed/{account_id} 检查JSONL
  3. 检查冷却期（5分钟）
  4. 设置前端执行锁
  5. 执行下单操作
  6. 下单成功 → POST /api/okx-trading/set-allowed/{account_id} 写入 allowed: false
  7. 释放前端执行锁

⚠️ 重要：即使多个浏览器/设备同时打开，JSONL文件锁确保只有第一个触发的请求能够执行。这是最可靠的防护层。
